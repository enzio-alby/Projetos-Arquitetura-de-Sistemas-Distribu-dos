# Sprint 2.1 — Especificação de Design (revisada 2026-05-09)

**Tópico:** Correções e lacunas da Sprint 2.1 — bug de REQUEUE, testes apontando para workerp2.py, cobertura de integração.

**Resumo/Objetivo:** Corrigir o bug crítico de REQUEUE multi-tarefa, atualizar os testes para cobrir `workerp2.py` (arquivo central), adicionar infraestrutura de testes faltante e adicionar teste de integração para o temp master servindo tarefas.

---

## Contexto do projeto

- Arquivos centrais: `masterp2.py` (servidor master, porta 7011), `workerp2.py` (worker com heartbeat, eleição, temp master).
- Arquivos legados (ignorar): `master/master.py`, `worker/worker.py`.
- Comunicação: TCP sockets com JSON terminado em `\n`.
- Protocolo: master lê **uma mensagem por conexão** e fecha — cada interação abre uma nova conexão TCP.

## Escopo das correções

### Correção 1 — Bug REQUEUE multi-tarefa (crítico)
`workerp2._requeue_tasks_to_master()` deve abrir uma **nova conexão TCP por tarefa**, respeitando o protocolo do master (uma mensagem por conexão). A implementação atual usa uma única conexão para N tarefas, o que falha silenciosamente a partir da segunda tarefa.

**Critério de aceitação:** Com 2 tarefas na fila do temp master, ambas chegam ao master original (verificado via teste de integração com fake master real — uma conexão por mensagem).

### Correção 2 — Imports dos testes (crítico)
Todos os testes unitários e de integração devem importar de `workerp2`, não de `worker.worker`. O arquivo `worker/worker.py` é legado e não é o arquivo central da sprint.

**Critério de aceitação:** `pytest tests/` passa importando apenas de `workerp2`.

### Correção 3 — Infraestrutura de testes
- `tests/conftest.py`: adiciona raiz do projeto ao `sys.path` automaticamente para pytest.
- `requirements-dev.txt`: lista `pytest` e `pytest-mock`.

**Critério de aceitação:** `pip install -r requirements-dev.txt && pytest tests/ -q` executa todos os testes.

### Correção 4 — Teste de integração: temp master servindo tarefas
Falta um teste que verifique que um worker consegue pegar uma tarefa do temp master e receber ACK. O único teste existente testa o requeue (retorno ao master original), não o ciclo normal de tarefa durante o período de temp master.

**Critério de aceitação:** Teste conecta como worker ao temp master, recebe `{"TASK": "QUERY"}`, envia `{"STATUS": "OK"}`, recebe `{"STATUS": "ACK"}`.

## Regras de compatibilidade

- Nenhuma mudança no protocolo TCP/JSON (mensagens existentes mantêm seus campos).
- `workerp2.py` e `masterp2.py` devem permanecer arquivos standalone (sem depender de `worker/worker.py` ou `master/master.py`).
- Testes devem rodar sem rede real — usar sockets locais (localhost) ou mocks.

## Arquitetura proposta

Manter a estrutura atual. As únicas mudanças de código são:
1. `workerp2._requeue_tasks_to_master()`: loop com `create_connection` dentro, não fora.
2. `tests/`: novos e corrigidos arquivos de teste apontando para `workerp2`.
3. `tests/conftest.py`: `sys.path` setup.
4. `requirements-dev.txt`: dependências de dev.

## Plano de validação

1. `pytest tests/unit/ -q` → todos passam.
2. `pytest tests/integration/ -q` → todos passam.
3. `python tests/integration/run_temp_master_requeue.py` → imprime `PASS`.
4. `python tests/integration/run_temp_master_serve.py` → imprime `PASS`.
